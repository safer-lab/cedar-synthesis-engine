---

## 1. Locked decisions

### 1.1 Atomization style (Q1)

**(b)-with-per-atom-review.** The agent proposes a complete draft of all
atoms for a stage, ordered for review. The user reviews each atom one
at a time with full context of the rest. Granularity is per-atom; the
user can approve, reject, edit, or question a single atom without
rewriting paragraphs.

### 1.2 Property atom scope (Q2)

**Five property atom types.** Three primitives are harness-facing; two
sugars compile down at plan-composition time. The harness
(`eval_harness.py`, `orchestrator.py`, `solver_wrapper.py`) sees only
ceiling/floor/liveness — sugars are invisible to it.

| Type | Role | Compiles to |
|------|------|-------------|
| `ceiling` | primitive | itself |
| `floor` | primitive | itself |
| `liveness` | primitive | itself |
| `rate_limit` | sugar | ceiling on `context.<counter> < N`, plus a Stage 1 schema atom for the counter (forces Stage 1.5) |
| `disjointness` | sugar | ceiling whose reference is `NOT(target_ref)`, plus automated §8.8 consistency patches on every floor |

Out of scope: `equivalence`, `cardinality`. The harness contract does
not change; sugars are local to `cedar_agent/property_elicitor.py`.

### 1.3 User skill assumption (Q3)

**Non-Expert primary; Expert as a power-user toggle.** Every atom has:

- a `plain_english_summary` field,
- an `examples_adversarial` list — examples generated by the symbolic
  analyzer to maximally distinguish the chosen encoding from plausible
  nearby alternatives the prose could also mean (see §4),
- a `reference_cedar` encoding hidden behind an `[S]ee Cedar` toggle.

The Expert toggle reveals the encoding inline. The atom dataclass
shape is the same in both modes; only the rendering differs.

### 1.4 What "verified" means in this system (load-bearing distinction)

The verifier (`cedar symcc`) discharges *formal* claims about the
encoding: satisfiability, joint consistency with prior atoms, type
correctness, and any universal claims a sugar makes (`disjointness`,
`rate_limit`). **It does not — and cannot — prove that the encoding is
a faithful translation of the prose.** That is a human judgment.

This is the load-bearing trust point of the architecture. The user
flow rests on the user reading the summary + adversarial examples and
authorizing the encoding as a faithful match for what they meant. The
UI must preserve this responsibility honestly:

- Badge text is `"✓ Formally consistent — does this match your intent?"`,
  not `"✓ Verified"`.
- Approval (`[A]`) sets a separate flag, `intent_acknowledged_by_user`,
  distinct from `symbolic_verified`. Both are recorded in the corpus.

§4 specifies how the symbolic analyzer earns the formal claims and
§4.4 specifies how adversarial examples are generated to make the
user's intent-verification step as diagnostic as possible.

### 1.5 Implied: Stage 1.5 exists (Q4)

The `rate_limit` sugar always needs a context-attribute counter that
the prose did not surface (it requires the host app to maintain it).
Because Stage 2 atoms can therefore force schema additions, **Stage 1.5
is in scope for Step B**: a controlled re-entry to Stage 1 triggered
by Stage 2 atoms whose compile-down requires new schema material.
Stage 1.5 is _agent-initiated and user-confirmed_; it is not a free
hand for the user to edit the schema during Stage 2.

## 3. Atom dataclasses (`cedar_agent/atoms.py`)

### 3.1 Common base

```python
@dataclass
class AtomBase:
    """Fields every atom carries."""
    name: str                          # stable identifier within the session
    rationale: str                     # short technical reason (always present)
    plain_english_summary: str         # primary review text
    source_excerpt: str                # the prose this atom was extracted from
```

### 3.2 Stage 1 — schema atoms

```python
@dataclass
class EntityAtom(AtomBase):
    members_of: list[str] = field(default_factory=list)
    attributes: dict[str, "AttributeAtom"] = field(default_factory=dict)
    enum_values: Optional[list[str]] = None

@dataclass
class AttributeAtom(AtomBase):
    on_entity: str                     # which Entity this attribute lives on
    field_name: str
    cedar_type: str                    # e.g. "Long", "Set<String>", "Record{...}"
    optional: bool = False
    alternatives_considered: list[str] = field(default_factory=list)

@dataclass
class ActionAtom(AtomBase):
    principal_types: list[str]
    resource_types: list[str]
    context_attributes: dict[str, AttributeAtom] = field(default_factory=dict)
    parent_groups: list[str] = field(default_factory=list)

@dataclass
class TypeAliasAtom(AtomBase):
    cedar_type: str                    # the body of `type X = ...`
```

### 3.3 Stage 2 — property atoms

```python
@dataclass
class Example:
    """One concrete request used in atom review.

    Adversarial examples (§4.4) are generated to distinguish the chosen
    encoding from plausible alternatives — every example is paired with
    one or more alternative readings of the prose, plus the actual
    decision under each. The user reads the divergence and authorizes
    the chosen encoding as the intended interpretation.
    """
    description: str                       # plain English ("Alice (nurse, on careTeam) viewing Bob's record")
    request_dict: dict                     # full cedar-evaluatable request
    decision_under_chosen: Literal["permit", "deny"]
    decisions_under_alternatives: dict[str, Literal["permit", "deny"]]  # alt label → decision
    diagnostic_for: list[str]              # which interpretive choices this example probes

@dataclass
class AlternativeEncoding:
    """One plausible alternative reading of an atom's prose."""
    label: str                             # e.g. "primary-nurse-only"
    interpretive_choice: str               # one-line description of what differs
    cedar_text: str

@dataclass
class PropertyAtom(AtomBase):
    constraint_type: Literal[
        "ceiling", "floor", "liveness",
        "rate_limit", "disjointness"
    ]
    action: str
    principal_types: list[str]
    resource_types: list[str]

    # Encoding (always populated; revealed only by Expert toggle).
    reference_cedar: str

    # Adversarial-within-encoding examples (§4.4).
    # These probe the boundary between the chosen encoding and the
    # plausible alternatives, so the user's eyeball-review is diagnostic.
    examples_adversarial: list[Example] = field(default_factory=list)
    alternatives_considered: list[AlternativeEncoding] = field(default_factory=list)

    # Symbolic verification (§4.1–§4.3). What symcc actually proved.
    symbolic_verified: bool = False
    symbolic_verification_log: list[str] = field(default_factory=list)
        # entries like "satisfiable: ok", "joint-consistency-with-atom-3: ok",
        # "type-correct: ok", "disjointness-claim-discharged: ok"

    # Intent verification (§1.4). Distinct from symbolic_verified.
    # Set True when the user clicks [A]pprove. The corpus log captures
    # both flags so we can later study where users approved despite
    # imperfect grounding.
    intent_acknowledged_by_user: bool = False

    # Stage 2.5 traceback (§7). Populated after Stage 3 converges.
    traceback_clauses: list[str] = field(default_factory=list)
        # candidate.cedar clause text(s) that discharge this atom
    traceback_flags: list[str] = field(default_factory=list)
        # silent-divergence flags per §6.2

    # Sugar-specific fields. Set only when constraint_type matches.
    rate_limit_window: Optional[str] = None        # "1m", "1h", etc.
    rate_limit_threshold: Optional[int] = None
    rate_limit_counter_attr: Optional[str] = None  # e.g. "requestsPerMinute"
    disjoint_with: Optional[str] = None            # name of the rule we're disjoint from
```

### 3.4 Composition objects

```python
@dataclass
class SchemaDraft:

```

## 4. Verification scaffolding (`cedar_agent/grounding.py`)

### 4.1 What symcc earns the verified badge for (and what it doesn't)

For every Stage 2 atom, before the atom is shown to the user, the
agent runs **four symbolic checks** via `cedar symcc`. The
`symbolic_verified: bool` flag is set True iff all four pass; the
`symbolic_verification_log` records which checks ran and their results.

1. **Satisfiability.** There exists a request the encoding permits
   (for `ceiling`/`floor`) or denies (for `disjointness`). Catches
   dead-rule encodings — a ceiling that permits nothing is technically
   sound but useless and almost certainly an error.

2. **Joint consistency.** The atom is jointly satisfiable with every
   previously-approved atom in the session. Catches the §8.8 trap at
   atomization time, not after Stage 3 fails to converge. For sugar
   atoms, this includes the §8.8 patches the sugar would force.

3. **Type correctness.** The atom validates against the composed
   schema (`cedar validate`). Free per-atom; we run it explicitly so
   schema-attribute typos surface at atomization, not later.

4. **Sugar-specific universal claims.** `disjointness` claims
   `∀req. R(req) ⇒ ¬target(req)`. `rate_limit` claims
   `∀req. context.<counter> ≥ N ⇒ deny`. Symcc proves or refutes these
   directly. Primitives (`ceiling`/`floor`/`liveness`) carry no sugar
   universal — their content is local and check 1–3 cover them.

### 4.2 What symcc does NOT earn the verified badge for

The four checks above prove the encoding has nice formal properties.
They do not prove the encoding is a faithful translation of the prose.
That is human judgment exercised by the user reading the summary and
the adversarial examples (§4.4). Per §1.4, this distinction is
preserved in the UI ("✓ Formally consistent — does this match your
intent?") and in the corpus (separate `symbolic_verified` and
`intent_acknowledged_by_user` flags).

### 4.3 API

```python
def symbolic_verify_atom(
    atom: PropertyAtom,
    schema_text: str,
    prior_atoms: list[PropertyAtom],
    cedar_path: str = CEDAR_PATH,
) -> SymbolicVerificationResult:
    """Run the four symcc checks. Mutates atom.symbolic_verified
    and atom.symbolic_verification_log. Does NOT decide whether the
    user should approve — that's the user's call after reading
    examples_adversarial."""
```

If any check fails, the agent re-prompts itself to revise the
encoding and re-runs verification. **The user never sees an atom
whose symbolic checks have failed.** Hard cap: 3 attempts per atom.
After 3 failures, the agent escalates with the failure mode visible
("I can't make this atom jointly consistent with atom #2; please
choose which to revise").

### 4.4 Adversarial-within-encoding examples

Generic illustrative examples ("Alice (hr) → Bob's review → permitted")
make for nice UI but are weak diagnostics. The user is unlikely to
catch a subtle interpretive error in the encoding by reading examples
that the encoding obviously satisfies. The grounding pipeline
generates examples that are **diagnostic by construction** — they
distinguish the chosen encoding from plausible nearby alternatives.

Pipeline per atom:

1. **LLM call: propose plausible alternatives.** Given the prose
   excerpt and the chosen encoding, propose 3 alternative encodings
   the prose could also be read as. Each alternative has an explicit
   `interpretive_choice` label (e.g. "primary-nurse-only" vs.
   "any-nurse-on-careteam").

2. **Symcc query: find a distinguishing request per alternative.**
   For each alternative `A`, query for a request `r` such that
   `chosen(r) ≠ A(r)`. If `chosen` and `A` are equivalent, the symcc
   query is unsat → drop that alternative (no diagnostic value).

3. **Format as Examples.** Each surviving distinguishing request
   becomes an Example whose `description` reads as
   `"<concrete actors> → chosen encoding: permit; alternative
   '<label>': deny"`. The user reads the divergence and authorizes
   the chosen encoding.

This is more involved than illustrative example generation but it
makes the user's intent-verification step diagnostic rather than
ceremonial. Per-atom cost: ~1 LLM call (alternative generation) plus
~3 symcc queries (one per alternative). Stays inside the per-session
cost envelope.

### 4.5 API for adversarial examples

```python
def propose_alternatives(
    atom: PropertyAtom,
    schema_text: str,
    n: int = 3,
) -> list[AlternativeEncoding]:
    """LLM call: given the prose excerpt and the chosen encoding,
    propose n plausible alternative readings of the prose, each with
    an explicit interpretive_choice label."""

def find_distinguishing_request(
    chosen: str,                       # chosen Cedar encoding
    alternative: AlternativeEncoding,
    schema_text: str,
    cedar_path: str = CEDAR_PATH,
) -> Optional[dict]:
    """Symcc query for a request where chosen and alternative
    disagree. Returns None if they're equivalent (no diagnostic
    value)."""

def generate_adversarial_examples(
    atom: PropertyAtom,
    schema_text: str,
) -> list[Example]:
    """Compose: propose alternatives → find distinguishers → format
    as Example records. Mutates atom.examples_adversarial."""
```

## 5. Sugar compile-down rules (`cedar_agent/property_elicitor.py`)

### 5.1 `rate_limit` → ceiling + schema amendment

```
PropertyAtom(
    constraint_type="rate_limit",
    action=A,
    rate_limit_window="1m",
    rate_limit_threshold=100,
    rate_limit_counter_attr="requestsPerMinute",  # or auto-generated
)
```

compiles to:

1. **Ceiling primitive** with reference
   `permit (...) when { context.<counter_attr> < <threshold> };`
2. **Schema amendment** (Stage 1.5): adds
   `<counter_attr>: Long` to the action's `context` shape if not
   already present. The agent prompts the user once to confirm the
   schema amendment if it didn't exist.

### 5.2 `disjointness` → ceiling with negated reference + §8.8 patches

```
PropertyAtom(
    constraint_type="disjointness",
    action=A,
    disjoint_with="legal_hold_rule",  # references another atom or a
                                      # named clause in source prose
)
```

compiles to:

1. **Ceiling primitive** with reference whose body is the negation of
   the target's body. If `target` is `permit (...) when { C };`, the
   ceiling is `permit (...) when { !C };` — meaning the candidate's
   permit set must be ⊆ {requests where !C}.
2. **§8.8 patches**: every existing `floor` atom on the same action
   gets `&& !C` appended to its `when`-clause. The user is shown the
   patch summary before approving the disjointness atom and may
   override individual patches.

### 5.3 Implementation note

Compile-down runs at `VerificationPlanDraft.compile()` time, not at
atom-proposal time. The atoms displayed to the user are the *named*
sugar atoms (rate_limit / disjointness). Only after all Stage 2 atoms
are approved does the compile step produce the primitive
`references/*.cedar` files the harness consumes.

## 6. HITL UI contract (`cedar_agent/ui/terminal.py`)

### 6.1 What the user sees per atom

Atom rendering blocks have a fixed shape:

```
[<atom-type-label> <i> of <N>]  <constraint or entity name>

  <plain_english_summary>

  Source excerpt: "<source_excerpt>"

  Adversarial examples (probing the boundary with plausible alternatives):
    Bob (nurse, on careTeam, NOT primary nurse) → reading Alice's record
      chosen encoding ("any-nurse-on-careteam"):  permit
      alternative ("primary-nurse-only"):         deny
    Bob (nurse, primary nurse, NOT on careTeam) → reading Alice's record
      chosen encoding:  deny
      alternative ("primary-nurse-only"):  permit
    ...

  <type-specific fields, e.g. for rate_limit:
   "Schema impact (Stage 1.5): adds context.requestsPerMinute: Long.">

  ✓ Formally consistent — does this match your intent?
    [satisfiable; jointly consistent with atoms 1, 2; type-correct;
     for disjointness atoms: universal claim discharged]

  [A]pprove  [R]eject  [E]dit  [Q]uestion  [S]ee Cedar
```

The `✓ Formally consistent` line is the verified badge from §1.4 —
explicitly framed as "what symcc proved" rather than "verified
correct." `[A]pprove` sets `intent_acknowledged_by_user = True` in
the corpus log; the symcc state was already logged.

For sugars that trigger §8.8 patches or schema amendments, an
additional `[V]iew patches / amendments` action is offered.

### 6.2 What the user can input per atom

| Key | Action | Agent response |
|-----|--------|----------------|
| `A` | Approve | Set `intent_acknowledged_by_user = True`. Lock atom, advance to next. |
| `R` | Reject + free-text reason | Agent proposes an alternative atom; loop on the new atom. The rejected atom and its alternatives are logged for the corpus. |
| `E` | Edit (inline) | User edits the summary, threshold, examples, or Cedar (with `[S]`). Agent re-runs symbolic verification AND regenerates adversarial examples before re-presenting. Approval not assumed; the revised atom must be re-shown and re-confirmed. |
| `Q` | Ask a question | Agent answers; atom is re-presented unchanged for a fresh decision. |
| `S` | Show Cedar encoding | Inline reveal of `reference_cedar`. Atom remains in the same review state. |
| `V` | View sugar patches / schema amendments | Inline reveal of the §8.8 patches or Stage 1.5 amendments this atom forces. Atom remains in the same review state. |

### 6.3 Stage 2.5 traceback rendering and `⚠` semantics

After Stage 3 converges, the user sees the atom-to-policy traceback
(per §7.4). Each approved atom is shown alongside the candidate
clause(s) that discharge it, plus a flag indicator:

```
Atom 4 (legal-hold disjointness)
  enforced by candidate.cedar lines 17-19:
    forbid (...) when { resource.legalHold && principal.role != "legal" };
  ✓ symcc confirms this clause discharges the disjointness obligation
```

`⚠` is reserved for **non-obvious clause-atom relationships only** —
specifically:

- The candidate clause(s) discharging this atom reference attributes
  not mentioned in the atom's prose excerpt.
- The atom's satisfaction depends on interaction between multiple
  clauses, not a single clause.
- The encoding uses a Cedar idiom the user did not encounter during
  atomization (e.g., a `forbid ... unless` where atomization showed
  only `permit ... when`, or set-algebra patterns where a role-string
  comparison was expected).

`⚠` is NOT used for "shorter form available" — that is critic
territory (§7.5), not traceback territory. Surfacing structural-but-
benign rewrite hints in 2.5 would train users to ignore the warnings,
defeating the purpose. Critic-driven refactor suggestions live in a
separate section of the post-synthesis output.

### 6.4 Stage transitions

After all Stage 1 atoms approved → run `cedar validate` on the
composed schema → if invalid, agent fixes and re-validates (max 3
attempts) → schema written to disk → Stage 2 starts.

After all Stage 2 atoms approved → run pre-Stage-3 unsat detection
(§7.3) → run sugar compile-down → write `verification_plan.py` and
`references/*.cedar` → Stage 3 invokes the unmodified v1 harness via
`run_scenario(scenario_dir)` with the critic loop layered on top
(§7.5) → Stage 2.5 traceback presented to user → final approval →
session closes.

## 7. Pipeline orchestration (`cedar_agent/pipeline.py`)

The pipeline now has six stages: 1, 1.5, 2, 1.75, 3 (with critic), 2.5.
Numbering follows the doc; ordering follows the call graph.

### 7.1 Top-level flow

```python
def author(spec_path: str, output_dir: str) -> Path:
    """End-to-end. Returns the path to the user-approved candidate.cedar."""
    spec_text = Path(spec_path).read_text()

    # Stage 1 — schema atomization
    atomizer = SchemaAtomizer(llm, verifier)
    schema_atoms = atomizer.propose_schema(spec_text)
    schema_draft = atomizer.review_loop(schema_atoms)
    schema_path = atomizer.synthesize_and_validate(schema_draft, output_dir)

    # Stage 2 — property elicitation
    elicitor = PropertyElicitor(llm, verifier, grounding)
    prop_atoms = elicitor.propose_properties(spec_text, schema_path)

    # Stage 1.5 — schema amendments forced by sugars
    amendments = elicitor.detect_schema_amendments(prop_atoms, schema_draft)
    if amendments:
        atomizer.add_amendments(amendments, schema_draft)   # user-confirmed
        schema_path = atomizer.synthesize_and_validate(schema_draft, output_dir)

    # Stage 2 review — symbolic verification + adversarial examples
    # per atom (§4); grounding sees the final post-amendment schema.
    plan = elicitor.review_loop(prop_atoms, schema_path)

    # Stage 1.75 — pre-synthesis unsat detection
    unsat_check = symbolic_consistency_check(plan, schema_path)
    if unsat_check.unsat:
        # Atoms are mutually inconsistent. Bounce to Stage 2 with the
        # unsat core so the user can resolve.
        plan = elicitor.resolve_unsat(plan, unsat_check.core)
        # Re-run Stage 1.75 after resolution. Loop bounded.

    elicitor.write_plan(plan, output_dir)

    # Stage 3 — synthesis with verifier + critic loop
    candidate_path = synthesize_with_critic_loop(
        scenario_dir=output_dir,
        phase2_model="claude-haiku-4-5-20251001",
        max_iters=20,
        quality_threshold=4.0,    # critic 1-5 scale
    )

    # Stage 2.5 — atom-to-policy traceback, final user approval
    traceback = generate_atom_traceback(plan, candidate_path, schema_path)
    final_ok = traceback_review_loop(traceback)
    if not final_ok:
        # User rejects the synthesized policy. Iterate.
        ...
    return candidate_path
```

State persistence lives in `cedar_agent/ui/persistence.py`: the
in-progress draft is written to `<output_dir>/.session.json` after
each user decision so the session can be resumed. Resume is mandatory
for the ~30-minute user-attention budget.

### 7.2 Stage 1.5 — schema amendment

A Stage 2 atom may require new schema material — most commonly a
`rate_limit` atom requiring a `context.<counter>: Long` attribute.
The pipeline detects amendments before review_loop runs grounding,
so adversarial examples are generated against the final schema.

User confirmation per amendment is single-click ("the rate_limit atom
needs `context.requestsPerMinute: Long`; add to the bulkExport
action's context? [Y/N]"). Rejection forces re-resolution of the
underlying Stage 2 atom (drop the rate_limit, encode it differently,
etc.).

### 7.3 Stage 1.75 — pre-synthesis unsat detection

After all Stage 2 atoms are approved (with both `symbolic_verified`
and `intent_acknowledged_by_user` set), the pipeline runs symcc on
the **conjunction of every reference policy** to verify the plan is
jointly satisfiable.

```python
def symbolic_consistency_check(plan, schema_path) -> ConsistencyResult:
    """For each pair of references on the same action, query symcc
    'is there a request both reference policies admit/forbid as
    expected?' Returns the unsat core (the smallest set of atoms
    whose joint constraint is unsat) when inconsistency is found."""
```

If a contradiction is found, the user sees the unsat core ("atoms #2
and #4 are jointly unsatisfiable on requests where principal is on
careTeam and the record is under legal hold") and resolves at the
atom level. **This catches the failure mode where the LLM iterates
forever on a satisfiable-looking but actually contradictory spec.**
The check is cheap — symcc can already do this; we just have to call
it.

If the plan is consistent, Stage 1.75 returns silently and Stage 3
proceeds.

### 7.4 Stage 2.5 — atom-to-policy traceback

After Stage 3 converges, the pipeline computes which candidate
clauses discharge which atoms. The procedure:

```python
def generate_atom_traceback(plan, candidate_path, schema_path):
    """For each approved atom A:
       1. Find the minimum subset of candidate clauses C ⊆ candidate
          such that C alone discharges A (verifier passes A on C).
       2. Compute silent-divergence flags per §6.3:
          - C references attributes not in A's source_excerpt
          - |C| > 1 (interaction between clauses)
          - Cedar idiom in C absent from A's encoding
       3. Populate atom.traceback_clauses and atom.traceback_flags.
    Returns the populated atom list for UI rendering."""
```

The user reviews the traceback (§6.3 rendering), inspects any `⚠`
flags, and gives a final approval. Rejection forces a re-iteration
of Stage 3 with feedback ("user rejected: atom #4's traceback used
unexpected attribute `resource.tenantId`; explain or revise").

### 7.5 Stage 3 — synthesis with verifier + critic loop

The synthesis loop is the unmodified v1 harness's CEGIS, augmented
with a quality gate. Each iteration:

1. LLM produces a candidate.
2. Verifier (`cedar symcc`) checks correctness (the existing
   orchestrator behaviour).
3. If verifier passes, the **critic** scores the candidate on
   intrinsic Cedar quality.
4. Termination: `verifier_passes AND critic_score >= threshold`,
   OR iteration budget exhausted.
5. On budget exhaustion: return the highest-critic-score verified
   candidate from the iteration history.

#### 7.5.1 Critic prompt boundary (load-bearing)

The critic must not see:
- The verification plan or any reference policies.
- The original prose spec.
- The verifier's iteration feedback.

The critic sees only:
- The candidate Cedar text.
- A generic Cedar style guide (idiomatic patterns, common anti-
  patterns, documentation style conventions).

Quality scoring is **intrinsic** — "is this idiomatic Cedar" — not
spec-relative. Spec-relative scoring would re-create the well-known
LLM-as-judge failure mode where the judge confuses "matches the spec"
with "is high quality." The verifier already owns spec-correctness;
the critic owns code quality, and their authority is non-overlapping.

#### 7.5.2 Critic dimensions

Score 1–5 on each, with one-sentence rationale:

- **Idiomatic**: uses Cedar patterns the way the docs recommend
  (attribute references vs. entity-ID enumeration, `unless` for
  exceptions, etc.).
- **Minimal**: no redundant when-conditions or unreachable clauses.
- **Attribute-prefer**: prefers attribute references over hardcoded
  entity IDs or string disjunctions where attributes exist.
- **Maintainable**: reasonable rule structure, optional `@id`/`@description`
  annotations, named action groups where present.

Composite threshold: mean ≥ 4.0 across dimensions, no single
dimension < 3. Tunable per-deployment.

The critic is **advisory, not blocking** in the verifier sense:
critic disagreement does not override verification. A verified-but-
low-critic candidate is still the fallback if the iteration budget
runs out. The verifier is the correctness oracle; the critic is the
quality oracle.

### 7.6 Out of scope: symbolic synthesizer fallback

A genuine symbolic synthesizer (Sketch/Rosette/Cedar-specific) as a
parallel path when the LLM fails to converge would be a stronger v2
contribution. For v1 the pre-Stage-3 unsat detection in §7.3 catches
the contradiction case; the LLM-iterates-forever-on-hard-but-sat-spec
case is rare and acceptable to handle with iteration budget + best-
candidate-by-critic-score fallback.

## 8. Out of scope for Step B

- `equivalence` and `cardinality` atom types.
- Web UI / TUI; terminal only.
- Multi-user collaborative review.
- Schema evolution across multiple sessions.
- Anything that touches `eval_harness.py`, `orchestrator.py`, or
  `solver_wrapper.py`. The harness contract is frozen.
- Symbolic synthesizer as a parallel fallback (per §7.6). Pre-Stage-3
  unsat detection covers the contradiction case; the LLM-stalls-on-

- Step F's full 15-input evaluation set; that is a separate

## 9. Acceptance criteria for Step B

Step B is done when:

1. **Atom dataclasses (§3).** `cedar_agent/atoms.py` defines all
   dataclasses listed, with pytest unit tests covering construction,
   sugar field validation, and JSON serialization for persistence.
2. **Symbolic verification (§4.1–§4.3).** `cedar_agent/grounding.py`
   runs the four symcc checks against a real `cedar` binary on a
   minimal handcrafted atom and correctly populates
   `symbolic_verified` and `symbolic_verification_log`.
3. **Adversarial examples (§4.4–§4.5).** Given a chosen encoding and
   a manually-constructed alternative, the grounding pipeline
   produces a distinguishing request via symcc and formats it as an
   Example. Tested on at least one atom per primitive type.
4. **Sugar compile-down (§5).** Produces byte-identical output to a
   reference Cedar text on five golden inputs — one per atom type
   including both sugars. The compile-down output also passes symcc
   joint-consistency (§7.3).
5. **Stage 1.75 unsat detection (§7.3).** Given a deliberately
   contradictory two-atom plan (handcrafted), `symbolic_consistency_check`
   returns the unsat core; given a satisfiable plan, returns ok.
6. **Stage 2.5 traceback (§7.4).** Given a synthesized candidate
   alongside its plan, `generate_atom_traceback` populates
   `traceback_clauses` for every atom and surfaces `⚠` flags only
   when one of the §6.3 silent-divergence conditions holds.
7. **Critic prompt boundary (§7.5.1).** The critic-prompt builder
   has a unit test asserting it does NOT include the verification
   plan, the spec, or the verifier feedback in the prompt sent to
   the LLM.
8. **Pipeline skeleton (§7.1).** `cedar_agent/pipeline.py` compiles
   and the test harness can call `author()` with stubbed LLM /
   `review_loop` callbacks end-to-end without errors. LLM stubs can
   be deterministic fixtures; real LLM integration is Step C/D.
9. **Corpus logging (§9.1).** The session log directory layout below
   is produced for a stubbed end-to-end run, including
   prose-excerpt-to-atom attribution as a first-class field. Per-atom
   logged events include `attribution_decision` and
   `alternative_attributions_considered`; per-decision logged events
   include `intent_acknowledged_by_user` distinct from
   `symbolic_verified`.
10. **Plan reference.** Every new file under `cedar_agent/` references
    this plan (`HITL_STEP_B_PLAN.md`) at the top.

When all ten criteria are met, Step B is signed off and Step C
(schema atomizer with real LLM integration) starts.

### 9.1 Corpus logging — directory layout

Every session writes a fully self-contained log under `eval_runs/`.
This is v1 work; the captured data will fund fine-tuning,
autocurriculum, and regression testing later (out of scope per §8).

```
eval_runs/<session-id>/
├── input/
│   └── policy_spec.md             # exact prose input, byte-preserved
├── transcript.json                # canonical flattened event log
│                                  # (timestamps + every decision in order)
├── stage1/
│   ├── proposed_atoms.json        # all atoms agent proposed for Stage 1,
│   │                              # including alternatives that lost
│   ├── attribution_decisions.json # per atom: which prose span the agent
│   │                              # claimed as source_excerpt + which
│   │                              # alternative spans it considered
│   ├── decisions.json             # per atom: user action (A/R/E/Q) +
│   │                              # free-text reasons + edit deltas
│   └── final_schema.cedarschema
├── stage1_5/
│   └── amendments.json            # schema amendments forced by Stage 2
│                                  # sugars; user confirmation per amendment
├── stage2/
│   ├── proposed_atoms.json
│   ├── attribution_decisions.json
│   ├── decisions.json
│   ├── symbolic_verification_logs.json   # per atom: symcc check results
│   ├── adversarial_examples.json         # per atom: alternatives proposed,
│   │                                     # distinguishers found, equivalents dropped
│   └── final_plan/                       # verification_plan.py + references/
├── stage1_75/
│   └── unsat_core.json            # if a contradiction was found and resolved;
│                                  # absent if plan was consistent on first try
├── stage3/
│   ├── iterations/
│   │   ├── iter_1/
│   │   │   ├── candidate.cedar
│   │   │   ├── verifier_feedback.json
│   │   │   └── critic_score.json   # critic's per-dimension scores + rationale
│   │   ├── iter_2/...
│   │   └── ...
│   └── final_candidate.cedar
├── stage2_5/
│   ├── traceback.json             # atom → clause(s) mapping + flags
│   └── final_user_decision.json   # final approval or rejection-with-reason
└── post_session_edits.json        # populated if/when the user later edits
                                   # the deployed policy; appended over time
```

**Critical fields the log must capture (these are the ones we can't
recover from the rest of the data):**

- For every atom: the prose excerpt the agent attached AND the
  alternative excerpts it considered before settling. This is the
  data needed to train a "prose-span → atom" attribution model later.
- For every user decision: `intent_acknowledged_by_user` separate
  from `symbolic_verified`. This lets us study (post-deployment) the
  cases where the user approved despite imperfect grounding — the
  most valuable training signal for improving the agent's
  prose-translation accuracy.
- For every Stage 3 iteration: both verifier feedback AND critic
  score, separately. This lets us correlate verifier-passing vs.
  critic-passing on the same iteration history and study how often
  they agree on convergence.

`transcript.json` is the canonical flattened event log — every
recorded event with a timestamp, in order. It is the easiest format
to grep, feed into fine-tuning, or replay as a regression test.
